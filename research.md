# 시스템 리서치 보고서: 드론 방황 및 작업 불능 현상 분석

## 1. 시스템 구조 및 동작 원리
현재 시스템은 `Mega_Crops.py`의 `mega_mixed_cultivation`을 중심으로 32개의 드론을 활용한 병렬 혼합 재배를 시도합니다. `functions.py`의 `for_all` 유틸리티를 통해 전체 필드 작업을 분산 처리하도록 설계되었습니다.

## 2. 주요 함수 및 레이어 관계
- **`for_all(f)`**: 드론을 생성하여 가로 방향(`row_task`) 작업을 맡기고 메인 드론은 세로로 이동하며 드론을 뿌립니다. 하지만 **생성된 드론의 종료를 기다리는 로직(`wait_for`)이 누락**되어 있습니다.
- **`mega_mixed_cultivation()`**: 초기화(`for_all(harvest)`) -> 워커 생성 -> 대기 -> 최종 수확 순으로 진행됩니다.
- **`mixed_worker_logic(x)`**: 특정 열을 담당하여 위로 올라가며 `get_companion()` 체인을 추적합니다.

## 3. 잠재적 위험 요소 (사이드 이펙트) 및 오류 원인

### 3.1 드론 자원 고갈 및 순차 실행 (Race Condition)
`for_all(harvest)`가 호출되면 즉시 32개의 드론이 생성되지만, 이 드론들이 작업을 끝내기 전에 메인 드론이 다음 단계인 `spawn_worker(x)`를 호출합니다.
- **결과**: 모든 `spawn_worker` 호출이 `None`을 반환하며 실패합니다.
- **사이드 이펙트**: 메인 드론이 `mixed_worker_logic`을 직접 수행하게 되는데, 이는 32x32 필드에 대해 단일 드론이 수천 번의 `go(tx, ty)`를 수행하는 비효율을 초래합니다.

### 3.2 필드 클리어 실패로 인한 무의미한 이동
`wait_for` 부재로 인해 필드가 수확되지 않은 상태에서 파종 로직이 시작됩니다.
- **결과**: `get_entity_type() == None` 조건이 충족되지 않아 `plant()` 함수가 호출되지 않습니다.
- **사이드 이펙트**: 드론은 아무것도 심지 못한 채, 기존 식물의 동반 식물 위치로만 이동을 반복하며 "방황"하는 것처럼 보이게 됩니다.

### 3.3 이동 로직의 중복성과 비효율
`mixed_worker_logic`이 담당 구역의 모든 타일을 시작점으로 5단계 체인 추적을 수행하므로, 이미 다른 드론이나 체인이 방문한 타일을 중복 방문하는 횟수가 매우 높습니다.

## 4. 해결 방안 제안
1. **`for_all` 수정**: 드론 핸들을 배열에 담아 반환하거나, 함수 내부에서 모든 드론이 끝날 때까지 기다리도록 수정해야 합니다.
2. **워커 로직 최적화**: 이미 식물이 심어진 타일은 체인 추적을 생략하거나, 방문 기록을 남겨 중복 이동을 줄여야 합니다.
3. **수확 조건 강화**: `harvest()` 호출 전 필드가 확실히 비워졌는지 확인하는 동기화 로직이 필요합니다.
